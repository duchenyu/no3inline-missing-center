## 1. R2 低斜率空性奇偶定理 —— ✅ 真定理，可入 README

**性质**：双侧解析证明 + n=5..60 穷举验证一致。
- 偶 n：中心方向两坐标皆奇 → 归约后仍两坐标奇 → 不可能落在 LOW（LOW 每方向至少一偶坐标）→ 平凡空。✓ 对所有偶 n 成立，无需数据。
- 奇 n≥5：竖直线三点 `(m+2,m−1),(m+2,m),(m+2,m+1)` 显式构造，中心方向 `(2,1),(1,0),(2,−1)` 全 ∈ LOW、连线 `(0,1)` ∈ LOW。✓

**评审**：这是四组里**唯一经得起"定理"标签**的结果。证明干净、双边、不依赖采样。

**README 写法建议**：作一条可证定理陈述，明确其范围是"辅助的中心方向超图 Hₙ^dir[L]"，并诚实说明它**不直接推进 NTIL 可解性**（只是揭示一个奇偶结构）。不要把它包装成"对主问题的进展"。

**精度更正**：定理原文"低斜率(L) 恒为 0"在 R1 的表里被复述，但奇偶定理只证**偶数 n** 空；奇数 n 恰**非空**。R1 表仅列偶 n（12,16,…,32）故 L=0 无矛盾，但任何"恒为 0"的泛化表述须补"偶数 n"限定。

## 2. R1 D₆ 主导是 bug 产物 —— ✅ 重要纠错（计算事实），可入 README

**性质**：
- `collinear3` 只遍历 `range(3)`（6 点中前 3）的 bug 已**确证**（用同一 bug 复算可得 94–100% D₆，正是原声称来源）。
- 校正后用线枚举法 + n=12 brute-force 交叉验证（4978 一致），得到真超边 D₆ 占比 63%→28%、高斜率 72% 主导。这是 **n=12..32 的计算事实**，正确。

**评审**：纠错本身有发表价值（自校正、防止错误结论传播）。数值对测试区间可靠。

**但须标注的两点**：
1. "D₆ 占比随 n→∞ 趋于 0" 是**猜想**（依据：D₆ 仅 6 个固定方向，真超边分散到 ~n² 方向）。合理但未证，不能写进 README 当定理。
2. 这些数值描述的是**辅助模型 Hₙ^dir 的真超边结构**，不是 NTIL 解的计数或可行性。其"主问题意义"有限，应降级为"对中心方向超图的校正性观察"。

**README 写法建议**：作"勘误 / 计算注记"段落，明确 (a) bug 已定位修复；(b) 校正后数值仅限 n≤32 实测、外推为猜想；(c) 这是辅助模型的性质。

## 3. R3 "相变假说死亡" —— ⚠️ 事实部分可入，强结论须删

**事实部分（真）**：`flammenkamp_cache` 中 n=58,60,62,64,66,68,70,72 的 `rot4.few` 一直存在（解数 19/32/5/25/2/2/1/1）。即 rot4 子类在 n≤72 的偶数上**连续有解**。

**过度宣称（须删/软化）**：
- 原文"相变假说目前零有效证据支撑，且有一条反证（稀疏化）"、"从数学上排除了硬性相变"。
- **稻草人风险**："相变/阈值"在本项目里始终只是**候选开放假说**（见 research_directions 备忘录 B 方向），从未被断言为事实；声称"已证伪"是把弱候选当靶子打。
- **范围错误**：rot4 解存在至 n=72，只否定了"rot4 计数在 n≈56 塌缩"这一**具体子声称**。它**完全不触及**核心开放问题——`D(n)=2n` 是否对所有 n 成立、n=71 是否有解。n=72 rot4 有解 ≠ "NTIL 易解"。
- **数据仍偏**：rot4 计数 19→1 单调递减，晚期（n=70,72 仅 1 解）恰说明该子类在收缩，"即将塌缩"并未被排除；只是下限被推到 >72。

**README 写法建议**：只写事实——"缓存数据表明 rot4 对称解在 n=12..72 的每个偶数上均存在（解数见附表），故该子类在此区间无计数真空。" 删去"相变死亡/被证伪"表述，改为主问题开放声明。

## 4. R4 2-因子冲突率 ∝ m^−1.68 —— ⚠️ 方法学硬伤，强结论无效

### 4.1 数值本身正确，但精度差
`c4_conflict_refined.py` 计算无误：rate(m)≈327.751·m^−1.681（m=10,14,18,22,26）。但这是 **5 个点的 log-log 拟合**：
- 指数 −1.68 的 95% CI 很宽（粗略约 [−1.3, −2.0]），因为 n=5、自由度仅 3、t 乘数 3.18。
- 唯一稳健结论是"成对冲突率随 m 下降"，**不是**精确的 −1.68 幂律。

### 4.2 致命的方法学缺陷（必须指出）
`orbits_conflict(i1,j1,i2,j2,n)` 只取**两个轨道（8 点）**做 C(8,3) 共线检测。
而 NTIL-C₄ 解的**真实可行性约束是三轨道共线**：一条网格线上三点分别来自**三个不同轨道**。
- 三不同轨道的共线三元组，在任意两个轨道的 8 点子集里**根本不出现** → `orbits_conflict` **完全漏检**这类真约束。
- `orbits_conflict` 只捕捉"两轨道退化型"（2 点同轨道 + 1 点另一轨道，多为过心线碰撞）。

故 `orbits_conflict` 是真实约束的 **必要非充分条件（代理变量）**：
- 它 = False 并不保证 2-因子无 3-in-line（仍可能有三轨道共线）；
- 真实约束图是 **3-均匀超图**（C(m,3)~m³ 个三元组），而它测的是 m²/2 个成对代理。

**因此**："成对代理冲突率稀疏化 → 从数学上排除相变/支持大 n 存在性" 这一推理链
**在逻辑上不成立**。稀疏的是代理，不是真实约束；且即便真实成对冲突→0，
也不能推出存在无冲突 2-因子（全局存在性需独立论证，如 LL / 稀疏超图方法）。

### 4.3 评审结论
- 脚本算的是真实数，但**测错了对象**。
- "∝ m^−1.68 排除相变" = **无效推论**，不能入 README。
- 可保留的只有：*"在两个轨道的退化型共线（过心线碰撞）意义上，轨道间冲突率随 m 下降"* —— 且须明确这是代理、非可行性约束、5 点拟合不精确。
